1. Security Layers Overview

Seven security zones from machine edge to application consumer

2. Layer 1 — Edge / Machine Security

Securing the first mile: machine CAN bus to Kafka broker

Telematics Unit (TU) Security

Each machine has a telematics unit that reads from the CAN bus, buffers data locally, and transmits over cellular. The TU is the trust root for machine identity.

ControlMechanismWhy
Device IdentityX.509 certificate per telematics unit, signed by CAT internal PKI. Certificate bound to machine serial number from MDM.Ensures only registered CAT machines can publish to Kafka topics. Prevents rogue device injection.
Key StorageHardware Security Module (HSM) or Trusted Platform Module (TPM) on the telematics unit. Private key never leaves hardware.Prevents key extraction even if physical device is compromised.
TransportMutual TLS (mTLS) from TU to Kafka broker. Both sides present certificates.Bidirectional authentication. Prevents man-in-the-middle attacks on cellular network.
Edge Buffer EncryptionAES-256 encryption on local buffer (used when cellular connectivity is lost). Key derived from HSM.Data-at-rest protection if device is physically stolen from a job site.
Firmware IntegrityOTA firmware updates signed by CAT code-signing certificate. TU validates signature before applying.Prevents malicious firmware updates from compromising telemetry or using device as attack vector.
Store and ForwardWhen connectivity drops, TU buffers up to 72 hours of telemetry locally (encrypted). Replays in order when connection restores.Data completeness guarantee. No silent data loss during remote site operations.
Threat: A stolen telematics unit used to inject false telemetry (e.g., artificially healthy engine readings to suppress maintenance alerts).
Control: Certificate revocation list (CRL) checked at broker. MDM de-registration immediately revokes Kafka ACL for that serial number.

3. Layer 2 — Kafka Broker Security

ControlMechanismDetail
AuthenticationSASL/SSL — machines use mTLS certificates; internal services (Magritte, pipeline jobs) use SASL/SCRAM-SHA-512Different auth mechanisms per client type. Machines present X.509, internal services use rotating SCRAM credentials managed by AWS Secrets Manager.
AuthorizationTopic-level ACLs. Each machine certificate can PRODUCE to its domain topics only (e.g., telemetry.engine.*). Magritte can CONSUME only, not produce.Least-privilege per client. A compromised machine certificate cannot write to other domains or consume data.
Encryption in TransitTLS 1.3 for all broker-to-broker and client-to-broker communicationForward secrecy. Cipher suites: TLS_AES_256_GCM_SHA384, TLS_CHACHA20_POLY1305_SHA256.
Schema ValidationConfluent Schema Registry enforces Avro/Protobuf schema on produce. Schema Registry itself requires OAuth token to register/evolve schemas.Malformed telemetry payloads rejected at broker before reaching Foundry. Schema evolution requires explicit approval.
Network IsolationKafka brokers in private VPC subnet. No public internet access. Magritte Connector reaches brokers via AWS PrivateLink.Attack surface limited to internal network only.
Dead Letter Queue SecurityDLQ topic restricted to ops team ACL only. Messages in DLQ retain original encryption.Failed messages not accessible to application consumers. Ops team investigates without exposing raw payload broadly.

4. Layer 3 — Palantir Foundry Security

ControlMechanismDetail
Encryption at RestAES-256 for all datasets (Bronze, Silver, Gold). Foundry manages encryption keys. Customer-managed keys (BYOK) available via AWS KMS for highest-sensitivity datasets.All telemetry data encrypted at storage layer. BYOK for PII datasets (operator identity, customer fleet data).
Data MarkingsFoundry IAM markings applied at dataset and Ontology object level. Enforced by Foundry kernel — cannot be bypassed at query time.A dealer user querying Machine objects only ever sees machines in their assigned territory. Marking filters rows before data leaves Foundry.
PII TokenizationOperator name, GPS coordinates (precise), customer contact data tokenized at Silver layer. Raw values in Bronze accessible only to Data Engineering role.Gold and Ontology layers expose tokenized or bucketed values (e.g., GPS geofence zone, not exact coordinates) to application consumers.
Foundry Audit LogEvery dataset read, write, and query logged with user identity, timestamp, and query parameters. Logs retained 13 months. Immutable.Full audit trail for compliance. Supports GDPR right-of-access queries ("what data have we processed about this operator?").
Code Repository AccessPipeline code (transforms) versioned in Foundry Code Repository. Only Data Engineering role can commit. All changes require code review and pass CI checks before merging.Prevents unauthorized modification of Bronze→Silver transforms that could corrupt or exfiltrate data.
Dataset SharingGold datasets shared to downstream consumers (NBO Engine, Member Portal API) via Foundry dataset references with read-only permissions. No direct Bronze/Silver access for application teams.Application teams see only curated Gold and Ontology views. Raw telemetry never exposed externally.

API Gateway Controls

ControlImplementation
JWT ValidationAPI Gateway validates JWT signature (RS256), issuer, audience, and expiry on every request. Invalid tokens rejected with 401 before reaching Machine Domain APIs.
Scope EnforcementJWT scopes checked at API Gateway. A customer token with machines:read cannot call the /workorders write endpoint even if they know the URL.
Rate LimitingPer client_id: Portal users — 100 req/min. NBO Engine — 500 req/min. Burst limit 2x for 30 seconds. 429 response with Retry-After header on breach.
WAF RulesAWS WAF: SQL injection, XSS, oversized payloads (>1MB), known bad IPs (AWS Managed Rules). Custom rules for machine serial number format validation.
mTLS InternalService-to-service calls (NBO Engine → Machine Domain APIs, Notification Service → SNS) use mTLS with internal CA-signed certificates. No bearer tokens for internal hops.
API Audit LogEvery API call logged: client_id, user_id (from JWT sub), endpoint, response code, latency. Shipped to CloudWatch + SIEM (Splunk/Datadog). Retained 13 months.

Customer A queries machines belonging to Customer BInformation DisclosureHigh (if not controlled)Foundry IAM data markings at Ontology level. Machine Domain APIs enforce customer_id scope from JWT claims.

Malicious Adobe Commerce webhook replays an old order to trigger duplicate NBO model updatesReplay AttackMediumWebhook HMAC includes timestamp. Foundry endpoint rejects payloads older than 5 minutes. Idempotency key on order_id.
Compromised NBO Engine client secret used to bulk-read all machine Gold dataElevation of PrivilegeMediumClient Credentials tokens TTL 15 min. Secret rotated every 30 days via Secrets Manager. Anomaly detection on bulk read patterns (>100 machines/min triggers alert).
Pipeline engineer modifies Bronze→Silver transform to exfiltrate raw telemetryInformation DisclosureLowCode Repository requires 2-person review + CI gate. Foundry audit log captures all dataset writes. SIEM alerts on unusual output dataset volumes.
GDPR data subject requests deletion of operator telemetryComplianceCertain (regulatory)Operator PII tokenized at Silver. Token → raw mapping table deletable independently. Foundry dataset TTL policies for Bronze raw data.

8. Compliance & Data Privacy

RegulationApplicabilityControls in Place
GDPREU machine operators, EU dealer employees. Any telemetry linked to an identifiable individual (operator GPS, login events).PII tokenization at Silver. Bronze raw data retained max 90 days. Right-to-erasure via token mapping deletion. Data Processing Agreements with Palantir and AWS. Foundry hosted in EU region for EU data.
CCPACalifornia-based customers and dealers.Opt-out mechanism for telemetry-linked recommendations. Data inventory maintained in Foundry catalog. Consumer request workflow via Member Portal.
SOC 2 Type IIPalantir Foundry is SOC 2 Type II certified. AWS is SOC 2 Type II. CAT's own controls reviewed annually.Annual third-party audit. Foundry and AWS audit reports available under NDA for customer due diligence.
Data ResidencySome markets (EU, potentially India, Brazil) require data to stay in-country.Foundry multi-region deployment. EU telemetry processed in EU Foundry instance. Cross-region data transfer prohibited by IAM marking on EU-tagged datasets.
OT / Industrial SecurityMachine firmware OTA updates, CAN bus signal integrity.Signed OTA firmware (code signing cert). CAN bus signals not exposed outside telematics unit. NIST CSF framework applied to OT/IT boundary.
