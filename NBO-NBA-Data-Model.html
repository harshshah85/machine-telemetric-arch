1. NBO vs NBA — Definition

TermFull FormPurposeDestination
NBO Next Best Offer Recommend the right parts to purchase based on machine wear and telemetry signals Adobe Commerce — pre-populated cart, ranked parts list
NBA Next Best Action Recommend the right service action to take — inspect, schedule, dispatch, alert Member Portal — action cards, dealer scheduling, push notifications

2. Decision Framework Overview

[Palantir Foundry — Gold Layer] Machine health scores Component wear % Active fault codes Usage patterns Historical replacement data Customer purchase history │ ▼ [NBO/NBA Scoring Engine — Foundry AIP] Step 1: Signal Collection — gather all telemetry inputs per machine Step 2: Urgency Scoring — score each signal against thresholds Step 3: Action Classification — NBO (parts) vs NBA (service) vs both Step 4: Ranking — prioritize by urgency + revenue impact Step 5: Enrichment — add part SKU, pricing, dealer availability Step 6: Output — write scored recommendations to Gold layer │ ├──────────────────────────────────┐ ▼ ▼ [NBO Recommendations] [NBA Recommendations] → Adobe Commerce (parts cart) → Member Portal (action cards) → Member Portal (parts widget) → Dealer CRM (service dispatch) → Email/push notification → Push notification (critical only)

4. Scoring & Urgency Model

Every telemetry signal is scored against thresholds to produce an urgency level. Urgency drives both the NBO ranking and the NBA action type.

Urgency Score Calculation

Urgency LevelScore RangeTrigger ConditionNBO ActionNBA Action
CRITICAL 85 — 100 Component failure imminent (< 50 hrs), active critical fault, safety event Immediate parts offer — top of portal, push notification Immediate service dispatch — push alert to operator + dealer
HIGH 60 — 84 Wear > 75%, estimated failure < 200 hrs, active warning fault Priority parts offer — prominent portal placement Schedule service within 2 weeks
MEDIUM 35 — 59 Wear 50-75%, approaching OEM service interval Recommended parts — standard portal placement Plan service next scheduled downtime
ROUTINE 0 — 34 Wear < 50%, preventive maintenance upcoming Suggested parts — email/newsletter No immediate action — monitor

Urgency Score Formula

urgency_score = ( (wear_percentage × 0.35) -- heaviest weight — direct wear signal + (fault_severity_score × 0.30) -- critical=100, warning=60, info=20 + (hours_remaining_score × 0.20) -- inverse: fewer hours = higher score + (health_score_inverse × 0.10) -- 100 - overall_health_score + (historical_failure_adj × 0.05) -- +bonus if this part failed before on same model ) hours_remaining_score: < 50 hrs → 100 50-200 → 75 200-500 → 40 > 500 hrs → 10 fault_severity_score: CRITICAL → 100 WARNING → 60 INFO → 20 NONE → 0

10. Closed Loop — Feedback into Foundry

Every order placed in Adobe Commerce and every service action taken feeds back into Foundry to improve future NBO/NBA recommendations.
Adobe Commerce Order Confirmed │ ▼ Webhook Foundry Pipeline: 1. Update WorkOrder object — parts_ordered [] 2. Update Component object — last_replaced_date, reset wear_% 3. Update NBO_Recommendation — status = ORDERED 4. Log to /gold/order_history — feeds purchase propensity model retraining │ ▼ NBO Model Retrained (weekly): New data: did customer buy? Which rank position? Output: improved propensity scores for next cycle Service Completed (dealer confirmation) │ ▼ Dealer system webhook Foundry Pipeline: 1. Update WorkOrder object — completion_date, technician, cost 2. Update Component object — reset health_score, wear_% 3. Update NBA_Recommendation — status = COMPLETED 4. Log to /gold/service_history — feeds failure prediction model retraining │ ▼ NBA Model Retrained (weekly): New data: was predicted failure accurate? How far in advance? Output: improved urgency scoring thresholds

12. NBO/NBA Serving Architecture — Precompute, Cache & Serve

Real-time NBO computation at serve time is not viable. The urgency scoring formula joins five Gold datasets per machine, applies suppression rules, ranks parts, and calls Adobe Commerce for live pricing — 3–5 seconds per machine under load. The solution is a batch-precompute + serve-from-cache pattern: recommendations are scored hourly in Foundry, stored in a pre-computed Gold dataset, surfaced through the Ontology, and served by the Machine Domain API in under 400ms. The only component that runs live at serve time is the Adobe Commerce pricing call for the top-ranked part.

Adobe Commerce Pricing Cache — Serve-Time Design

Why only the top-ranked part gets a live pricing call: ├─ Calling Adobe Commerce for all 5 recommended parts = 5 × ~150ms = 750ms │ This blows the p95 < 400ms SLO. ├─ Users almost never scroll past the #1 recommendation without acting. └─ Remaining 4 parts are priced lazily — pricing call fires when user expands the "View More" section, not on initial page load. Pricing cache design (in-memory, Machine Domain API): Key: {sku}:{customer_group_id} TTL: 15 minutes Size: Top 500 SKUs × 5 customer group tiers = 2,500 cache entries Miss: Live Adobe Commerce call (~150ms). On timeout (>500ms): serve "Price available at checkout" — never block page load on pricing. Warm: Cache pre-warmed on API startup with top-100 SKUs × all tiers.

Year 3 — Event-Sourced On-Demand Re-Score

In Year 3, a CRITICAL fault event on the streaming path (Engine or DPF domain) triggers an immediate on-demand NBO re-score for that specific machine — bypassing the hourly batch for urgent cases. This closes the 0–60 minute gap where a customer with a failing engine sees stale NBO recommendations.
Year 3 — Critical Fault → On-Demand NBO Re-Score (target: <30 sec end-to-end) [Kafka event: fault_code=CRITICAL, domain=ENGINE] │ ▼ Magritte Kafka connector (streaming) [/bronze/engine_telemetry_raw] — new partition written │ ▼ Streaming path (Engine/DPF only) Ontology Alert object updated (within 90 sec per SLO) │ ▼ Foundry event trigger on new CRITICAL Alert object On-demand NBO scoring job fires for this serial_number only ├─ Reads latest Gold data for this machine (single-machine query) ├─ Runs urgency formula (scope: one machine, not full fleet) ├─ Writes updated row to /gold/nbo_recommendations (overwrite for this serial) └─ Ontology NboRecommendation object updated │ ▼ Machine Domain API (next request for this serial) Portal renders updated HIGH-urgency recommendation └─ Customer sees: "Engine fault detected — part replacement recommended now" Year 1/2: this path does not exist. Full hourly batch only. Year 3: on-demand path runs alongside hourly batch — batch remains the source of truth for all non-critical domains.
