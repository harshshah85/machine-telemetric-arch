1. Foundry Architecture Overview

All telemetric domains follow a three-layer data organization inside Palantir Foundry, flowing from raw ingestion through to business-ready objects consumed by the Member Portal, NBO/NBA Engine, and Adobe Commerce.
LayerNameDescriptionDataset Path
BRONZERaw LayerData exactly as received — no transformations, no cleanup. Source of truth for replay./foundry/data/bronze/{domain}_raw
SILVERCleaned LayerDeduplicated, validated, typed correctly, enriched with machine/customer metadata./foundry/data/silver/{domain}_cleaned
GOLDBusiness-Ready LayerAggregated, scored, and joined. What the portal, NBO/NBA engine and APIs consume directly./foundry/data/gold/{domain}_enriched

Data Flow

[Machine Sensors / ECM] │ ▼ [Kafka Topics] ──► Foundry Magritte Connector (streaming) [Legacy ETL] ──► Foundry Dataset Ingestion (batch) │ ▼ BRONZE: /foundry/data/bronze/ │ ▼ Pipeline Builder transforms SILVER: /foundry/data/silver/ │ ▼ Aggregation + scoring + ontology mapping GOLD: /foundry/data/gold/ │ ▼ [Ontology Objects] ──► [Machine Domain APIs] ──► [Member Portal] ──► [NBO/NBA Engine] ──► [Adobe Commerce]

2. Data Refinement, Ownership & Enrichment

Data moves through three layers in Foundry. Each layer has a distinct owner, toolset, and responsibility. Understanding who does what — and what contracts exist between teams — is critical to the architecture.

Team Ownership by Layer

LayerOwnerFoundry ToolsResponsibility
BRONZEData Engineering TeamMagritte connectors, batch ingestion pipelinesLand raw telemetry exactly as received — no changes, no cleanup
BRONZE → SILVERData Engineering TeamPipeline Builder, Code Workbook (PySpark)Clean, validate, deduplicate, normalize units, enforce schema
SILVER → GOLDData Modeling / Architecture TeamPipeline Builder, Code Workbook, Foundry AIPAggregate, score, join, enrich, write business-ready objects
GOLD → OntologyData Modeling / Architecture TeamFoundry Ontology ManagerMap Gold datasets to Object Types, define Link Types and properties

Bronze → Silver: What Data Engineering Does

Raw Bronze data arrives (from Kafka stream or batch ETL) │ ▼ Pipeline Builder applies the following transforms: 1. NULL CHECKS — Missing critical fields flagged (not dropped — preserved for audit) — e.g. engine_hours = null → record flagged, written to /silver/flagged 2. RANGE VALIDATION — Sensor readings outside physical limits rejected — e.g. engine_temp_c = -9999 → invalid, flagged — e.g. battery_voltage_v = 0 → invalid during operation, flagged 3. DEDUPLICATION — Same sensor ping received twice (Kafka at-least-once delivery) — Keep latest by timestamp, drop exact duplicates 4. UNIT NORMALIZATION — All temperatures → Celsius — All pressures → kPa — All distances → kilometers — All volumes → liters 5. TYPE ENFORCEMENT — string "87.4" → float 87.4 — Unix epoch → ISO 8601 Timestamp 6. TIMESTAMP ALIGNMENT — All timestamps converted to UTC — Machine local time offset stored separately 7. SCHEMA ENFORCEMENT — Records not matching expected schema → rejected to /silver/rejected — Schema version tracked per machine model (CAT 320 vs D6 differ) │ ▼ Silver dataset: clean, validated, typed, deduplicated

Silver → Gold: What the Data Modeling Team Does

Silver cleaned data │ ▼ Pipeline Builder + Code Workbook (PySpark) transforms: 1. AGGREGATION — Roll up time-series into period summaries e.g. 1000 engine_temp readings/day → daily avg, min, max, std_dev — Cumulative totals e.g. daily fuel_consumed → cumulative_fuel_consumed_ltrs 2. HEALTH SCORE CALCULATION (per domain) — Weighted formula per component type engine_health_score = ( oil_pressure_score × 0.30 + coolant_temp_score × 0.25 + engine_hours_score × 0.25 + fault_code_score × 0.20 ) — Overall machine health = weighted avg of all component scores 3. WEAR % CALCULATION — Compare actual hours/cycles vs. OEM rated lifespan e.g. track_shoe_wear_% = (engine_hours / oem_lifespan_hours) × 100 — Reset to 0 when WorkOrder confirms part replacement 4. ENRICHMENT JOINS (see enrichment section below) 5. ML MODEL OUTPUT MERGING — Foundry AIP runs failure prediction models weekly — Outputs written to /gold/ml_outputs — Merged into Machine and Component objects: predicted_failure_days, nbo_propensity_score, nba_urgency_score 6. ONTOLOGY OBJECT CREATION — Gold datasets mapped to Machine, Component, Alert objects — Link Types resolved (Machine HAS_COMPONENT, Machine TRIGGERS_ALERT) │ ▼ Gold dataset: scored, aggregated, enriched, ontology-ready

Where Enrichment Runs — Inside Foundry, Not External Jobs

All enrichment runs inside Foundry — there are no external Python jobs, custom servers, or Airflow DAGs. Foundry is a self-contained data platform with its own compute engine, scheduler, and code execution environment. The only exception is Adobe Commerce pricing, which is a live REST API call made at serve time by the Machine Domain API.

Foundry Tools Used — Summary

Schema Contract Between Teams

The Data Modeling team defined the schema contracts that Data Engineering had to conform to when delivering Silver datasets. This meant: field names, data types, units, and null-handling rules were specified upfront. If the Silver output didn't match the contract, the Silver → Gold pipeline would reject it — preventing bad data from reaching the portal or NBO engine.

3. Code Repository — Engineering Deep Dive

Foundry Code Repository is the production-grade tool for all enrichment pipelines. It is a Git-backed environment where transforms are written in Python (or Java/Scala), version-controlled, built via Foundry CI, and executed on managed Spark compute. Pipeline Builder is used for simple joins and filters; Code Repository is used for everything complex.

Repository Structure

my-telemetry-transforms/ ├── transforms-python/ │ ├── src/ │ │ └── mypackage/ │ │ ├── __init__.py │ │ └── transforms/ │ │ ├── bronze_to_silver/ │ │ │ ├── engine_clean.py │ │ │ ├── dpf_clean.py │ │ │ └── hydraulics_clean.py │ │ ├── silver_to_gold/ │ │ │ ├── engine_health.py │ │ │ ├── nbo_scoring.py │ │ │ └── urgency_score.py │ │ └── reference/ │ │ └── mdm_sync.py │ ├── conda_recipe.yaml ← Python dependencies │ └── build.gradle ← Foundry build config └── README.md Each .py file = one transform function (one input dataset → one output dataset). Multiple transforms can share utility functions via __init__.py imports.

@transform_df vs @transform — When to Use Which

DecoratorWhat It ReturnsWhen to UseLimitation
@transform_dfReturns a Spark DataFrame — Foundry writes the output dataset automaticallyMajority of transforms. Clean, concise. Best for full-recompute pipelines.Cannot do incremental processing — always reads full input dataset
@transformLow-level — gives you TransformInput and TransformOutput objects, full control over read/writeIncremental processing (read only new partitions), custom write modes, file-level access, or when you need multiple output datasetsMore verbose. Requires explicit output.write_dataframe(df) call.

@transform_df — Standard Full Recompute

from transforms.api import transform_df, Input, Output from pyspark.sql import functions as F @transform_df( output=Output("/silver/engine_telemetry_cleaned"), raw=Input("/bronze/engine_telemetry_raw"), mdm=Input("/reference/mdm_machine_registry"), ) def clean_engine_telemetry(raw, mdm): """Bronze → Silver: clean, validate, join MDM keys.""" return ( raw # Step 1: Drop structural nulls (no serial = unusable row) .filter(F.col("machine_id").isNotNull()) # Step 2: Deduplicate — take latest event per machine per minute .withColumn("_rank", F.row_number().over( Window.partitionBy("machine_id", F.window("event_ts", "1 minute")) .orderBy(F.col("event_ts").desc()) )) .filter(F.col("_rank") == 1).drop("_rank") # Step 3: Normalize units (raw values are mixed °F/°C, psi/kPa) .withColumn("coolant_temp_c", F.when(F.col("temp_unit") == "F", (F.col("coolant_temp_raw") - 32) * 5 / 9 ).otherwise(F.col("coolant_temp_raw"))) .withColumn("oil_pressure_kpa", F.when(F.col("pressure_unit") == "PSI", F.col("oil_pressure_raw") * 6.895 ).otherwise(F.col("oil_pressure_raw"))) # Step 4: Join MDM to resolve canonical serial number .join(mdm.select("machine_id", "serial_number", "machine_model"), on="machine_id", how="left") # Step 5: Quarantine rows with no MDM match (serial stays null → filtered downstream) .withColumn("mdm_resolved", F.col("serial_number").isNotNull()) )

@transform — Incremental Processing (Engine/DPF Streaming Path)

from transforms.api import transform, Input, Output, incremental @incremental() @transform( output=Output("/silver/engine_streaming_cleaned"), source=Input("/bronze/engine_telemetry_raw"), ) def process_incremental(ctx, source, output): """ Incremental mode: only process partitions added since last build. Foundry tracks the high-water mark — no custom state management needed. """ if ctx.is_incremental: # Read only newly-added data (Foundry resolves the delta automatically) new_data = source.dataframe(ctx, "added") else: # First build or full recompute — read everything new_data = source.dataframe(ctx) cleaned = new_data.filter(F.col("machine_id").isNotNull()) # ... transform logic ... output.write_dataframe(cleaned)

4. Pipeline Builder — Visual Transform Tool

Pipeline Builder is Foundry's visual, no-code transform editor. You connect nodes to build a data pipeline. Each node is a discrete operation (filter, join, aggregate, etc.). Output is a Foundry dataset. No Spark code required — Foundry compiles the visual graph to a Spark execution plan at runtime.

Node Type Reference

NodeWhat It DoesSQL EquivalentWhen to Use in This Platform
SourceReads a Foundry dataset as input to the pipelineFROMEvery pipeline starts with at least one Source node
FilterDrops rows that don't match a condition (column comparisons, null checks, regex)WHEREQuarantine routing — e.g., route rows where mdm_resolved = false to the quarantine dataset
Select / RenameChoose which columns to keep; rename columnsSELECT col AS aliasTrim wide Bronze datasets before joining — reduces shuffle cost
ComputeAdds new columns using expressions — arithmetic, string functions, conditionalsSELECT ..., expr AS new_colSimple derived columns: wear_pct = engine_hours / oem_lifespan_hours * 100
JoinInner / left / right / cross join between two datasets on a keyJOIN ... ONEnrichment joins — Silver + MDM reference, Silver + OEM specs. Left join when right-side miss is acceptable (not for MDM key resolution).
AggregateGROUP BY — sum, count, avg, min, max, first, lastGROUP BY ... HAVINGDaily rollup of engine hours per machine; fault code frequency per domain

DeduplicateRemove duplicate rows, keeping first or last by a sort keyROW_NUMBER() OVER ... = 1Bronze deduplication — Kafka at-least-once delivery produces duplicate events; dedup by (machine_id, event_ts)

5. Schema Contracts & Data Quality Expectations

Schema contracts are the formal agreements between teams on what a dataset must contain — field names, types, units, null rates, and value ranges. In this platform, the Data Modeling / Architecture team defines contracts for each Silver and Gold dataset. Data Engineering must produce Silver output that passes the contract before the Silver → Gold pipeline runs. Contracts are enforced via Foundry Expectations — code-level data quality checks that block downstream builds on failure.

What a Schema Contract Defines

Contract ElementExample (Engine Silver)Enforced By
Field namesmachine_serial_number, event_ts, engine_hours, coolant_temp_c, oil_pressure_kpa@transform_df output schema check at build time
Data typesserial_number: StringType, event_ts: TimestampType, engine_hours: DoubleTypeFoundry schema validation on dataset write
Units (explicit)Temperature in °C, pressure in kPa, hours as cumulative float — no mixed units permitted in SilverContract document + Expectation range checks
Null ratesserial_number: 0% nulls (hard block); engine_hours: < 1% nulls allowed; fault_code: nulls expected (no active fault = null)Foundry Expectations (is_not_null, null_rate checks)
Value rangesEngine hours: 0–50,000; Coolant temp: -40°C to 140°C; Oil pressure: 0–700 kPaFoundry Expectations (between, greater_than checks)
UniquenessOne row per (serial_number, event_ts) — no duplicatesFoundry Expectations (is_primary_key check)

Contract Versioning & Breaking Changes

Schema changes are treated as breaking if they remove a field, rename a field, change a type, or tighten a null constraint. Breaking changes require an Architecture Decision Record (ADR) and a migration plan before they can be merged.
Change TypeBreaking?Process
Add a new nullable columnNoPR to Code Repo → CI → merge. Downstream transforms ignore unknown columns.
Rename a columnYesADR required. Maintain old column name alongside new for one release cycle. Update all downstream transforms before removing old name.
Remove a columnYesADR required. Confirm no downstream transform reads the column (check lineage graph). Two-step: deprecate (add warning), then remove.
Change a column type (e.g., String → Double)YesADR required. High-risk: Foundry Ontology object IDs must be stable — a type change on a primary key requires a full dataset reload and Ontology re-population.
Tighten a null constraintYesData Engineering must backfill historical data to meet the tighter constraint before the Expectation is deployed.
Change a value unit (e.g., °F → °C)YesADR required. All downstream consumers must be updated simultaneously — staged rollout not possible for unit changes.

Contract Governance — Who Owns What

TeamContract Responsibility
Data Modeling / ArchitectureAuthors the Silver and Gold schema contracts. Approves all breaking changes via ADR. Owns the Foundry Ontology schema.
Data EngineeringImplements Bronze → Silver transforms that must satisfy the Silver contract. Flags when source data makes a contract field impossible to meet (triggers a contract revision, not a silent violation).
NBO Engine / Portal teamConsumers of Gold datasets. Must not build against undocumented fields — only fields in the Gold contract are guaranteed stable. Reports contract violations to Data Modeling team, does not work around them.
Architecture teamApproves all ADRs for breaking schema changes. Owns the cross-team schema governance process.
