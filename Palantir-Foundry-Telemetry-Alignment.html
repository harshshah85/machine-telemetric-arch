1. Foundry Architecture Overview

All telemetric domains follow a three-layer data organization inside Palantir Foundry, flowing from raw ingestion through to business-ready objects consumed by the Member Portal, NBO/NBA Engine, and Adobe Commerce.
LayerNameDescriptionDataset Path
BRONZERaw LayerData exactly as received — no transformations, no cleanup. Source of truth for replay./foundry/data/bronze/{domain}_raw
SILVERCleaned LayerDeduplicated, validated, typed correctly, enriched with machine/customer metadata./foundry/data/silver/{domain}_cleaned
GOLDBusiness-Ready LayerAggregated, scored, and joined. What the portal, NBO/NBA engine and APIs consume directly./foundry/data/gold/{domain}_enriched

Data Flow

[Machine Sensors / ECM] │ ▼ [Kafka Topics] ──► Foundry Magritte Connector (streaming) [Legacy ETL] ──► Foundry Dataset Ingestion (batch) │ ▼ BRONZE: /foundry/data/bronze/ │ ▼ Pipeline Builder transforms SILVER: /foundry/data/silver/ │ ▼ Aggregation + scoring + ontology mapping GOLD: /foundry/data/gold/ │ ▼ [Ontology Objects] ──► [Machine Domain APIs] ──► [Member Portal] ──► [NBO/NBA Engine] ──► [Adobe Commerce]

2. Data Refinement, Ownership & Enrichment

Data moves through three layers in Foundry. Each layer has a distinct owner, toolset, and responsibility. Understanding who does what — and what contracts exist between teams — is critical to the architecture.

Team Ownership by Layer

LayerOwnerFoundry ToolsResponsibility
BRONZEData Engineering TeamMagritte connectors, batch ingestion pipelinesLand raw telemetry exactly as received — no changes, no cleanup
BRONZE → SILVERData Engineering TeamPipeline Builder, Code Workbook (PySpark)Clean, validate, deduplicate, normalize units, enforce schema
SILVER → GOLDData Modeling / Architecture TeamPipeline Builder, Code Workbook, Foundry AIPAggregate, score, join, enrich, write business-ready objects
GOLD → OntologyData Modeling / Architecture TeamFoundry Ontology ManagerMap Gold datasets to Object Types, define Link Types and properties

Bronze → Silver: What Data Engineering Does

Raw Bronze data arrives (from Kafka stream or batch ETL) │ ▼ Pipeline Builder applies the following transforms: 1. NULL CHECKS — Missing critical fields flagged (not dropped — preserved for audit) — e.g. engine_hours = null → record flagged, written to /silver/flagged 2. RANGE VALIDATION — Sensor readings outside physical limits rejected — e.g. engine_temp_c = -9999 → invalid, flagged — e.g. battery_voltage_v = 0 → invalid during operation, flagged 3. DEDUPLICATION — Same sensor ping received twice (Kafka at-least-once delivery) — Keep latest by timestamp, drop exact duplicates 4. UNIT NORMALIZATION — All temperatures → Celsius — All pressures → kPa — All distances → kilometers — All volumes → liters 5. TYPE ENFORCEMENT — string "87.4" → float 87.4 — Unix epoch → ISO 8601 Timestamp 6. TIMESTAMP ALIGNMENT — All timestamps converted to UTC — Machine local time offset stored separately 7. SCHEMA ENFORCEMENT — Records not matching expected schema → rejected to /silver/rejected — Schema version tracked per machine model (CAT 320 vs D6 differ) │ ▼ Silver dataset: clean, validated, typed, deduplicated

Silver → Gold: What the Data Modeling Team Does

Silver cleaned data │ ▼ Pipeline Builder + Code Workbook (PySpark) transforms: 1. AGGREGATION — Roll up time-series into period summaries e.g. 1000 engine_temp readings/day → daily avg, min, max, std_dev — Cumulative totals e.g. daily fuel_consumed → cumulative_fuel_consumed_ltrs 2. HEALTH SCORE CALCULATION (per domain) — Weighted formula per component type engine_health_score = ( oil_pressure_score × 0.30 + coolant_temp_score × 0.25 + engine_hours_score × 0.25 + fault_code_score × 0.20 ) — Overall machine health = weighted avg of all component scores 3. WEAR % CALCULATION — Compare actual hours/cycles vs. OEM rated lifespan e.g. track_shoe_wear_% = (engine_hours / oem_lifespan_hours) × 100 — Reset to 0 when WorkOrder confirms part replacement 4. ENRICHMENT JOINS (see enrichment section below) 5. ML MODEL OUTPUT MERGING — Foundry AIP runs failure prediction models weekly — Outputs written to /gold/ml_outputs — Merged into Machine and Component objects: predicted_failure_days, nbo_propensity_score, nba_urgency_score 6. ONTOLOGY OBJECT CREATION — Gold datasets mapped to Machine, Component, Alert objects — Link Types resolved (Machine HAS_COMPONENT, Machine TRIGGERS_ALERT) │ ▼ Gold dataset: scored, aggregated, enriched, ontology-ready

How Each Enrichment Is Built & Run

Kafka handles the arrival of raw telemetry into Foundry. Enrichment happens after it lands — in scheduled Foundry pipelines. They are separate concerns.
Kafka Foundry ───────────────── ────────────────────────────────── Machine sensors Magritte Connector │ (real-time stream) ──► lands data into Bronze dataset │ │ │ ▼ │ Pipeline Builder: Bronze → Silver │ (every 15 minutes) │ │ │ ▼ │ Pipeline Builder: Silver → Gold │ (enrichment joins + scoring, hourly) │ │ │ ▼ │ Foundry AIP: ML model outputs merged │ (weekly retraining cycle) │ │ │ ▼ └────────────────────────── Machine Domain APIs serve updated data to portal / NBO / NBA

Where Enrichment Runs — Inside Foundry, Not External Jobs

All enrichment runs inside Foundry — there are no external Python jobs, custom servers, or Airflow DAGs. Foundry is a self-contained data platform with its own compute engine, scheduler, and code execution environment. The only exception is Adobe Commerce pricing, which is a live REST API call made at serve time by the Machine Domain API.

Schema Contract Between Teams

The Data Modeling team defined the schema contracts that Data Engineering had to conform to when delivering Silver datasets. This meant: field names, data types, units, and null-handling rules were specified upfront. If the Silver output didn't match the contract, the Silver → Gold pipeline would reject it — preventing bad data from reaching the portal or NBO engine.

4. Domain: Fuel & Emissions (DPF / DEF)

DPF (Diesel Particulate Filter) — Captures soot particles from diesel exhaust to meet emissions standards. Requires replacement or regeneration when soot load exceeds threshold.
DEF (Diesel Exhaust Fluid) — Urea-based fluid (also called AdBlue) injected into exhaust stream to reduce NOx emissions. Must be refilled regularly.
