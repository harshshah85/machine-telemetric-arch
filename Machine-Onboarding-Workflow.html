Machine Onboarding Workflow

How a New Machine Model Gets Registered into the Telemetry Platform

Prepared by Harsh Shah  |  May 2026

1. Why Onboarding is an Architectural Problem

The current architecture assumes machines already exist in MDM and are already streaming telemetry. In reality, CAT releases new machine models annually, dealers take delivery of new units, and customers add machines to their fleets constantly. Without a governed onboarding workflow, each new machine type becomes a manual engineering project — the platform doesn't scale.

Two distinct onboarding scenarios:
A. New machine model (e.g., CAT 395 Next Gen Excavator) — new telemetry signals, potentially new CAN bus mappings, new schema registration required. Happens 2–4x per year. Engineering-led.
B. New machine unit (existing model, new serial number) — machine physically exists, needs to appear in MDM and start streaming. Happens hundreds of times per year. Largely automated.

2. Scenario A — New Machine Model Onboarding

Engineering-led process. Estimated effort: 2–4 weeks per new model family.

1
Signal Discovery & CAN Bus Mapping Manual — OEM Engineering OEM Engineering provides the J1939/CAN bus signal dictionary for the new model. Each signal maps to: parameter group number (PGN), signal name, unit, range, sample rate, and criticality. This document is the source of truth for schema design.
2
Schema Design & Registration Data Modeling Team Data Modeling team translates the CAN bus signal dictionary into an Avro/Protobuf schema. Schema registered in Confluent Schema Registry under the appropriate domain topic namespace (telemetry.engine.v2, etc.). Schema version bumped — backward compatibility validated (new fields must be optional/nullable). Schema contract published to Foundry Code Repository as a checked-in specification file.
3
MDM Registration MDM Team (External) New machine model added to MDM: model number, year, engine type, component list, warranty terms, OEM spec thresholds (wear limits, maintenance intervals). MDM is the external system — Data Engineering raises a ticket to MDM team. MDM sync runs the next day, populating /bronze/mdm_machine with the new model's spec data.
4
Magritte Connector Configuration Data Engineering Magritte connector updated to consume the new Kafka topic (or new schema version on existing topic). Landing Bronze dataset path confirmed: /bronze/{domain}_{model_family}_raw. Connector deployed to Foundry dev environment first. Smoke test with 100 synthetic events.
5
Bronze→Silver Transform Update Data Engineering Existing Silver transforms updated (or new transform added) to handle new signals. New fields added to Silver schema. Data Quality Checks updated: new null rate checks, range checks for new signals. Deployed to Foundry dev, validated against synthetic data, then promoted to staging via code review gate.
6
Gold Layer & Ontology Property Updates Data Modeling Team If new signals introduce new health metrics or component properties, Gold aggregation logic updated. New Ontology properties added to Component or Machine object types (additive only — no breaking changes). Existing consumers unaffected by additive property additions (Foundry Ontology supports this by design).
7
NBO Parts Mapping Parts Catalog Team New model's components mapped to Adobe Commerce SKUs in the parts catalog Silver dataset. NBO suppression rules reviewed — any new components that need warranty suppression or order-history suppression configured. NBO Engine tested with synthetic urgency scenarios for new model.
8
Telematics Unit Provisioning OEM Manufacturing / Dealer Telematics unit on the physical machine provisioned with X.509 device certificate (PKI workflow). Certificate bound to machine serial number from MDM. Kafka ACL created for the serial number. Machine is now authorized to stream telemetry.
9
End-to-End Validation & Sign-off Platform Engineering 10-machine pilot: stream live telemetry from 10 units of the new model. Validate Bronze landing, Silver cleanliness, Gold health scores, Ontology object creation, NBO recommendation generation, and Member Portal display. Sign-off checklist completed by Platform Engineering, Data Engineering, and Product. Gate to production onboarding of full model fleet.

3. Scenario B — New Machine Unit Onboarding (Existing Model)

Largely automated. Target: machine streaming to Foundry within 48 hours of dealer delivery.

5. Year 3 Vision — Self-Service Machine Onboarding Portal

By Year 3, the onboarding workflow should be accessible to authorized users without engineering involvement for Scenario B.

Self-Service Portal Capabilities (Year 3 Target)

  • Dealer submits new machine registration via Member Portal (serial number, model, delivery date)
  • Portal validates against MDM API in real-time — confirms serial is known
  • Automated certificate provisioning triggered (no Data Engineering ticket)
  • Dealer receives confirmation: "Machine CAT-EX-2024-00142 is active and streaming. First health assessment available within 24 hours."
  • OEM Internal team gets new machine coverage dashboard update
  • If serial not in MDM: portal surfaces MDM registration form — dealer submits basic machine details, MDM team gets automated approval workflow
This is the difference between a platform and a project. When onboarding a new machine requires a Jira ticket to Data Engineering, the platform doesn't scale. Self-service onboarding is the architectural goal — it means the platform is truly a product, not an internal consulting service.
