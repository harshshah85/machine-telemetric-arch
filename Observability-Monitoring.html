1. The Three Pillars of Observability

Metrics — What is happening right now?

Quantitative time-series data: pipeline throughput, API latency, error rates, cache hit ratio, NBO conversion rate. Collected every 60 seconds. Stored in CloudWatch + Datadog. Used for SLO burn-rate alerting and capacity planning.

Logs — Why did something happen?

Structured JSON logs from every component: Magritte connector, pipeline transforms, Machine Domain APIs, NBO Engine, webhook receiver. Centralized in Datadog Log Management or AWS OpenSearch. Retained 90 days. Searched for root cause after an incident.

Traces — Where did a request go and what slowed it down?

Distributed tracing via AWS X-Ray or Datadog APM. Trace ID propagated from Member Portal → API Gateway → Machine Domain APIs → Foundry Ontology query. Identifies latency hotspots across service boundaries.

DiagnoseCheck metrics dashboard for anomaly start time. Pull correlated logs in Datadog. Trace specific failing requests via X-Ray.Platform EngineerDatadog + X-Ray
MitigateApply fastest available fix (rollback deployment, increase pipeline compute, clear Redis cache, re-run failed pipeline job).On-call engineerFoundry admin + AWS console
ResolveConfirm SLO metrics return to green. Notify stakeholders. Close incident channel.Incident commanderPagerDuty resolve + Slack
Post-mortemBlameless post-mortem within 48 hours for all P1s. 5-whys root cause. Action items tracked in Jira with owners and deadlines.Incident commander + teamConfluence + Jira
Blameless culture note: Post-mortems focus on system and process failures, not individual mistakes. The goal is to make the system more resilient — not to assign blame. This principle is non-negotiable and modeled by leadership.
